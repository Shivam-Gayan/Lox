# Abstract Syntax Tree (AST)

Welcome to the **AST (Abstract Syntax Tree)** module of the Lox interpreter. This directory contains the structural representation of Lox source code. 

Once the `Parser` successfully validates the grammar of the tokens generated by the `Scanner`, it organizes them into this hierarchical, tree-like data structure. The AST represents the logical flow and operations of the code, stripped of irrelevant syntax like whitespace and redundant parentheses.

## Core Architecture: Expressions vs. Statements

The Lox language grammar is strictly divided into two primary categories, represented by the two abstract base classes at the root of this folder:

### 1. `Expr` (Expressions)
An **Expression** is any piece of code that evaluates to a *value*. If you can assign it to a variable or print it, it's an expression.
* **Base Class:** `Expr.cs`
* **Nodes Location:** `/Expressions` directory.
* **Examples:** `1 + 2` (Binary), `"hello"` (Literal), `myFunc()` (Call), `!true` (Unary).

### 2. `Stmt` (Statements)
A **Statement** is an instruction that performs an action or alters the state of the program but does *not* evaluate to a value. 
* **Base Class:** `Stmt.cs`
* **Nodes Location:** `/Statements` directory.
* **Examples:** `var x = 5;` (Var), `print x;` (Print), `if (true) {}` (If), `while(true) {}` (While).

## The Visitor Pattern (The Secret Sauce)

If you look at the AST node classes (like `Binary.cs` or `While.cs`), you will notice they contain almost no logic—they are purely data transfer objects (DTOs) holding properties. 

**Why?** Because an AST needs to be traversed by multiple different systems:
1. The **Interpreter** needs to evaluate it.
2. The **Resolver** needs to analyze variables and lexical scopes.
3. An AST Printer might need to print it for debugging.

If we added an `Interpret()`, `Resolve()`, and `Print()` method to every single AST node, the classes would violate the **Single Responsibility Principle** and become a nightmare to maintain.

Instead, we use the **Visitor Pattern**:
1. `Expr.cs` and `Stmt.cs` define an interface: `IVisitor<R>`.
2. Every visitor (like our `Interpreter`) implements this interface, defining a specific method for each node type (e.g., `VisitBinaryExpr(Binary expr)`).
3. Every AST node implements a single abstract method: `public override R Accept<R>(IVisitor<R> visitor)`.
4. When a visitor wants to process a node, it calls `node.Accept(this)`. The node then turns around and calls the correct, strongly-typed method on the visitor: `return visitor.Visit[NodeType](this);`.

This elegant **double-dispatch** mechanism completely decouples the tree structures from the algorithms that operate on them.

## 📂 Directory Breakdown

### `/Expressions`
This folder contains all the nodes that evaluate to values:
* **`Assign.cs`**: Modifying an existing variable (`a = 5`).
* **`Binary.cs`**: Standard two-operand operations (`a + b`, `a * b`, `a << b`).
* **`Call.cs`**: Function or class instantiation calls (`myFunc()`, `Person()`).
* **`Get.cs`**: Accessing a property on an object (`instance.property`).
* **`Grouping.cs`**: Parentheses to force precedence `(a + b)`.
* **`Literal.cs`**: Raw values (`123`, `"string"`, `true`, `nil`).
* **`Logical.cs`**: Short-circuiting boolean operators (`and`, `or`).
* **`Set.cs`**: Assigning a value to an object's property (`instance.property = value`).
* **`Super.cs`**: Accessing superclass methods (`super.method()`).
* **`This.cs`**: The `this` keyword for current instance context.
* **`Unary.cs`**: Single-operand prefix operations (`!true`, `-5`, `~10`).
* **`Variable.cs`**: Accessing a declared variable (`x`).

### `/Statements`
This folder contains all the nodes that control execution flow and environment state:
* **`Block.cs`**: A scoped sequence of statements enclosed in `{ ... }`.
* **`Break.cs`**: Immediately exits the current loop (`break;`).
* **`Class.cs`**: Defines a new class, its superclass, and its methods (`class A < B { ... }`).
* **`Continue.cs`**: Jumps to the next iteration of the current loop (`continue;`).
* **`Expression.cs`**: A statement that exists solely to evaluate an expression (e.g., a standalone function call `doWork();`).
* **`Function.cs`**: Defines a function, its parameters, and its body (`fun add(a, b) { ... }`).
* **`If.cs`**: Conditional branching (`if ... else`).
* **`Print.cs`**: Evaluates an expression and outputs it to the console (`print x;`).
* **`Return.cs`**: Returns a value from a function (`return value;`).
* **`Var.cs`**: Declares a new variable in the current scope (`var x = 1;`).
* **`While.cs`**: A standard loop structure (`while (condition) { ... }`). 
  * *Note: `for` loops are desugared (syntactic sugar) by the Parser into `While` nodes and `Block` nodes, so there is no dedicated `For.cs` node!*
